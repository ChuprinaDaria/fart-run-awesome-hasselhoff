# Fartrun Dev Health — Специфікація Checks

> Система моніторингу, оптимізації та менторінгу для вайбкодерів.
> Не пише код. Не запускає агентів. Сканує, показує, пояснює, зупиняє.

---

## Підтримувані стеки (v1)

- **Python** (.py, requirements.txt, pyproject.toml, Pipfile)
- **JavaScript/TypeScript** (.js, .ts, .jsx, .tsx, package.json)

## Формат реалізації

- **Rust** — парсинг файлів, пошук по AST, великі кодові бази (швидкість критична)
- **Python** — git-аналіз, dependency checks, генерація звітів, tips engine

## Фаза 1 — Project Map ("Що в тебе взагалі є")

> Мета: дати людині карту її власного проєкту. Більшість вайбкодерів не знають що в них лежить і де.

### Check 1.1 — File Tree Summary

| Поле | Значення |
|------|----------|
| ID | `map.file_tree` |
| Мова реалізації | Rust |
| Що робить | Сканує проєкт, будує дерево з статистикою |
| Вивід | Кількість файлів по типу, розмір, глибина вкладеності |
| Підказка | "В тебе 847 файлів. З них 312 — node_modules. Це нормально, але знай що вони є." |

### Check 1.2 — Entry Points Detection

| Поле | Значення |
|------|----------|
| ID | `map.entry_points` |
| Мова реалізації | Rust |
| Що робить | Знаходить точки входу: main.py, index.js, app.py, manage.py, package.json scripts |
| Вивід | Список entry points з коротким описом що кожен робить |
| Підказка | "Точка входу — це файл з якого все починається. Як двері в будинок. В тебе їх 3." |

### Check 1.3 — Module/Component Map

| Поле | Значення |
|------|----------|
| ID | `map.modules` |
| Мова реалізації | Rust |
| Що робить | Парсить import/require/from, будує граф залежностей між файлами проєкту |
| Вивід | Які файли від яких залежать, які є "хабами" (їх імпортує багато хто) |
| Підказка | "utils.py імпортується з 14 файлів. Це твій найважливіший модуль. Зламаєш — зламається все." |

### Check 1.4 — Monster File Detection

| Поле | Значення |
|------|----------|
| ID | `map.monsters` |
| Мова реалізації | Rust |
| Що робить | Знаходить файли > 500 рядків (warning), > 1000 (high), > 3000 (critical) |
| Вивід | Список файлів-монстрів з кількістю рядків, функцій/класів всередині |
| Підказка | "app.py — 3241 рядок і 47 функцій. Це не файл, це роман. Один файл = одна відповідальність." |

### Check 1.5 — Config & Env Inventory

| Поле | Значення |
|------|----------|
| ID | `map.configs` |
| Мова реалізації | Python |
| Що робить | Знаходить всі конфіг-файли: .env, .toml, .yaml, .json configs, dockerfile, docker-compose |
| Вивід | Список конфігів, де лежать, які env vars використовуються |
| Підказка | "В тебе 3 файли .env в різних папках. Це хаос. Зазвичай потрібен один .env в корені." |

## Фаза 2 — Dead Code ("Що з цього мертве")

> Мета: знайти код який нікому не потрібен. Менше коду = менше багів = менше плутанини.

### Check 2.1 — Unused Imports

| Поле | Значення |
|------|----------|
| ID | `dead.unused_imports` |
| Мова реалізації | Rust |
| Що робить | Знаходить import/require які імпортуються але ніде не використовуються у файлі |
| Вивід | Файл, рядок, що імпортовано |
| Підказка | "Невикористаний імпорт — це як купити продукти і забути в багажнику. Видали, нічого не зламається." |

### Check 2.2 — Unused Functions/Classes

| Поле | Значення |
|------|----------|
| ID | `dead.unused_definitions` |
| Мова реалізації | Rust |
| Що робить | Знаходить def/function/class які визначені але ніде не викликаються в проєкті |
| Вивід | Файл, рядок, назва, тип (function/class) |
| Підказка | "Функція calculate_discount() існує з коміту від лютого. Ніхто її не викликає. Або видали, або використай." |

### Check 2.3 — Orphan Files

| Поле | Значення |
|------|----------|
| ID | `dead.orphan_files` |
| Мова реалізації | Rust |
| Що робить | Файли .py/.js/.ts які ніхто не імпортує і які не є entry points |
| Вивід | Список файлів-сиріт |
| Підказка | "old_api.py — ніхто не імпортує, не entry point, лежить з березня. Якщо треба — перенеси в archive/. Якщо ні — видали." |

### Check 2.4 — Commented-Out Code Blocks

| Поле | Значення |
|------|----------|
| ID | `dead.commented_code` |
| Мова реалізації | Rust |
| Що робить | Знаходить великі блоки закоментованого коду (>5 рядків підряд) |
| Вивід | Файл, рядки, preview |
| Підказка | "42 рядки закоментованого коду в routes.py. Це не backup, для цього є git. Видали." |

### Check 2.5 — Duplicate Code Blocks

| Поле | Значення |
|------|----------|
| ID | `dead.duplicates` |
| Мова реалізації | Rust |
| Що робить | Знаходить блоки коду (>10 рядків) які повторюються в різних файлах |
| Вивід | Пари файлів, рядки, % збігу |
| Підказка | "Цей блок з 15 рядків є в api.py і в views.py. Винеси в окрему функцію, імпортуй з обох." |

## Фаза 3 — Tech Debt ("Що горить")

> Мета: знайти проблеми які зараз не болять, але вибухнуть пізніше.

### Check 3.1 — Outdated Dependencies

| Поле | Значення |
|------|----------|
| ID | `debt.outdated_deps` |
| Мова реалізації | Python |
| Що робить | Перевіряє pip/npm залежності на наявність оновлень і відомих вразливостей |
| Вивід | Пакет, поточна версія, остання версія, severity якщо є CVE |
| Підказка | "requests 2.28.0 → є 2.32.3. Оновлення зазвичай безболісне. Чим довше чекаєш — тим більше зламається при оновленні." |

### Check 3.2 — Missing Type Hints / JSDoc

| Поле | Значення |
|------|----------|
| ID | `debt.no_types` |
| Мова реалізації | Rust |
| Що робить | Знаходить функції без type hints (Python) або без JSDoc/TS типів |
| Вивід | Файл, функція, кількість параметрів без типів |
| Підказка | "process_data(x, y, z) — що таке x? Тип підказує і тобі, і AI. def process_data(x: list[dict], y: str, z: int) — тепер зрозуміло." |

### Check 3.3 — Error Handling Gaps

| Поле | Значення |
|------|----------|

| Підказка | "except: pass — це як вимкнути пожежну сигналізацію. Помилка є, ти її просто не бачиш." |

### Check 3.4 — Hardcoded Values

| Поле | Значення |
|------|----------|

| Що робить | Знаходить magic numbers, hardcoded URLs, порти, timeouts, розміри |
| Вивід | Файл, рядок, значення |
| Підказка | "sleep(300) — що таке 300? Секунд? Мілісекунд? Зроби TIMEOUT_SECONDS = 300 і використовуй константу." |

### Check 3.5 — TODO/FIXME/HACK Audit

## Фаза 4 — Brake System ("Стій, подумай")

| Вивід | Список незакінчених задач з файлами |
| Підказка | "В тебе 3 змінені файли не закомічені і 2 тести падають. Може спочатку це, а потім нові фічі?" |

## Severity Levels (для всіх checks)

| Level | Коли | Приклад |
|-------|------|---------|
| 💀 Critical | Це зламає все прямо зараз | 3000-рядковий файл, 0 тестів, .env в git |
| 🔴 High | Це вибухне скоро | Застарілі залежності з CVE, 15 зламаних тестів |
| 🟡 Medium | Це уповільнює роботу | Мертвий код, дублікати, відсутні типи |
| 🔵 Low | Гарна практика | TODO старіші 30 днів, великі коміти |
| ℹ️ Info | Просто знай | Кількість файлів, структура проєкту |

## Принципи

1. **Не пиши код за людину** — покажи що не так, поясни чому, дай команду для фіксу
2. **Контекстні підказки** — не лекція на 30 хвилин, а 2 речення коли вони потрібні
3. **Без менторського тону** — факти + конкретна дія, без "ти мусиш зрозуміти важливість..."
